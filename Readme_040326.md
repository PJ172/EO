### 1. Sửa lỗi params khi edit user (Next.js 16)
- **Vấn đề:** Khi bấm "Chỉnh sửa" một user, form bị render ở trạng thái "Thêm mới" và không load được dữ liệu cũ.
- **Nguyên nhân:** Project sử dụng Next.js phiên bản 16, yêu cầu `params` truyền vào Page Component phải là một `Promise` và dùng `await` để lấy giá trị. Code cũ sử dụng params đồng bộ `params: { id: string }` khiến `id` trả về `undefined`.
- **Giải pháp:** Cập nhật file `apps/web/src/app/(dashboard)/settings/users/[id]/edit/page.tsx` chuyển thành async function `await params` để nhận đúng ID.

### 3. Sửa lỗi ValidationPipe tại tầng API
- **Vấn đề:** Backend (NestJS) không báo lỗi 401 Unauthorized khi gửi request thiếu/sai mật khẩu admin mà vẫn cho phép bypass (vượt qua) Validation.
- **Nguyên nhân:** File `main.ts` của Backend chưa bật tính năng Global Validation Pipe khiến class-validator decorators (`@IsString`, `@MinLength`, v.v) trên UpdateUserDto hoàn toàn mất tác dụng.
- **Giải pháp:** Thêm `app.useGlobalPipes(new ValidationPipe({ whitelist: true, transform: true }));` vào file bootstrap `main.ts`. Mật khẩu admin tại server nay đã bị bắt buộc dài 6 ký tự trở lên. 
- Đồng thời bổ sung **Message Tiếng Việt** cho các validators error trả ra từ API trên `users.dto.ts`.

### 4. Xử lý "Bẫy Lỗi" API (API Client Interceptor)
- **Vấn đề:** Khi nhập sai mật khẩu admin và API trả về `401 Unauthorized`, giao diện bỗng dưng bay ra thẳng màn hình `/login` (ép logout) thay vì bắn chữ đỏ (Toast).
- **Nguyên nhân:** File Client xử lý API `api-client.ts` mặc định bắt mọi mã lỗi 401 trên HTTP và gắn nhãn là "Hết hạn phiên đăng nhập nên phải logout". 
- **Giải pháp:** Lập trình thêm ngoại lệ (whitelist) để khi gọi update (PATCH) user endpoint, lỗi 401 sẽ trả về đúng Exception của nó để màn hình Form hứng lỗi và render đỏ.

### 5. Ràng buộc Mật Khẩu Admin (Check ngay tại Bước Thông tin - Step 1)
- **Vấn đề:** Màn hình phân làm 2 bước (Thông tin / Phân Quyền). Nếu mật khẩu admin sai, phải chọn xong hết Quyền ở bước 2 rồi Submit nó mới báo lỗi khiến người dùng bị ức chế vì thao tác rườm rà.
- **Giải pháp:** 
  - Tại API Backend (`users.controller.ts` & `users.service.ts`): bổ sung thêm endpoint `POST /users/verify-admin-password` giúp kiểm tra độc lập mật khẩu.
  - Tại Frontend (`user-form.tsx`): Chuyển logic của nút "Tiếp Theo" ở Step 1 thành bất đồng bộ (Async). Nó sẽ bắn trước dữ liệu mật khẩu Admin lên Backend kiểm tra bằng endpoint mới.
    - Nếu đúng -> Nhảy sang Bước 2.
    - Nếu sai -> Form giữ nguyên ở Bước 1 và bắn lỗi Đỏ ngay lập tức.
    - Thêm Icon Quay đều (loader spinner) khi đang thao tác kiểm duyệt.

### 6. Thùng rác phân tán (Decentralized Trash) cho Module Nhân sự
- **Vấn đề:** Hệ thống Thùng rác tập trung hiện tại yêu cầu người dùng phải chuyển sang một module riêng biệt để kiểm tra/phục hồi dữ liệu, gây gián đoạn luồng làm việc (Work flow).
- **Giải pháp - Backend:** 
  - Thêm tính năng Lọc Dữ liệu Đã xóa (`isDeleted`) vào truy vấn danh sách (`GET /employees`).
  - Thêm/chuyển các endpoints Khôi phục (`POST /employees/:id/restore`) và Xóa Vĩnh Viễn (`DELETE /employees/:id/force`) vào trực tiếp service của module Nhân sự.
- **Giải pháp - Frontend:**
  - Chuyển đổi giao diện trang Danh sách Nhân sự (`hr/employees`) thành giao diện có TABS (Đang làm việc / Thùng rác).
  - Viết logic React cho UI để vô hiệu hóa/chuyển đổi tính năng của các nút Hành Động giữa chế độ làm việc và chế độ thùng rác (VD: Ẩn nút Chỉnh Sửa, bật Mở Khôi phục).
  - Tích hợp thêm Confirm Dialogs phòng chống bấm nhầm xoá vĩnh viễn. Đảm bảo hỗ trợ chọn nhiều dòng (Bulk Actions) và hiển thị lưới Thẻ nhân viên (Grid view).
  - Tối ưu hiệu năng: Call API riêng biệt theo tab, giảm Payload tải về lần đầu.
